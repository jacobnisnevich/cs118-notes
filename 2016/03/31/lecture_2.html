Data Delivery Performance

  • 3 basic measurements
    • Delay (sec, msec)
    • Throughput (bits/sec, Kbps, Mbps, Gbps)
    • Loss rate (% of packets that did not make it)
  • Transmission delay: L / R
    • R = link bandwidth (bps)
    • L = packet length (bits)
  • Propagation: d / s
    • d = length of physical link
    • s = propagation speed in medium (~2 x 10^8 m/sec)
  • Queueing delay = # packets in queue x transmission time of each packet

Lets Try an Example

  • A —- R —-> S1 ——– 2R ——–> S2 —- R —-> B
  • L = 4000 bits, R = 2 Mbps, A sends 4 packets
    • L / R = 2 ms, L / 2R = 1 ms
  • Propagation delay: 2 ms for each link
  • When will the first packet get to S2?
    • A – S1: D_transmission + D_propagation = 2 ms + 2 ms
    • S1 – S2: D_transmission + D_propagation = 1 ms + 2 ms = 3 ms
  • When the first packet arrives at S2, where is the second packet?
    • This occurs at t = 7 ms
    • At t = 4 ms, last bit leaves A
    • At t = 6 ms, last bit arrives at S1
    • At t = 7 ms, last bit leaves S1

A summary: Where is the Packet?

  • A ————> B
    • Bandwidth: W,
    • Link length: 200 km
    • Packet size: 1000 bits
  • Q: when the first bit of the packet reaches B, where is the last bit?
    • P = propagation delay = 200,000 m / 2 x 10^8 m/sec = 1 ms
  • If transmission delay = P (W = 1 Mbps)
  • If transmission delay = 0.1 P
    • bandiwdth * propagation delay = amount of data in the pipe

Client-server application communication model

  • Servers:
    • Reachable by IP address
    • Always-on, waiting for incoming requests from clients
  • Clients:
    • Initiate communication with server
  • Q: How does a client process identify the server process with which it wants to communicate?
  • A: using port numbers via the socket API

A quick comment about “port”

  • Web, email, ftp all use TCP
  • How does the server tell who wants what?
    • By port number: web uses port 80, ftp uses 21, and email uses 25

Socket

  • Process: program running on a host
  • Between different hosts: processes communcate through an application-layer protocol
  • A process sends/receives messages to/from its socket
  • Socket analogous to door:
    • Sending process shoves message out of the door
    • Transport protocol brings message up to the socket from the process

What is “Socket”

  • A set of system function calls
    • socket() - create a socket
    • bind() - bind a socket to a local IP address and port #
    • connect() - initiating connection to another socket
    • listen() - passively waiting for connections
    • accept() - accept a new connection
    • write() - write data to a socket
    • read() - read data from a socket

Establishing a socket on the client side

  • Create a socket with the socket() system call
  • Connect the socket to the server using the connect() system call
  • Send and receive data
    • There are a number of ways to do this, but the simplest is to use the read() and write() system calls

Establishing a socket on the server side

  • Create a socket with the socket() system call
  • Bind the socket to [address, port #] using the bind() system call
  • Listen for connections with the listen() system call
  • Accept a connection with the accept() system call
    • This call typically blocks until a client connects with the server
  • Send and receive data

Web and HTTP

  • Web page: normally consists of
    • Base HTML-file, which includes several referenced objects
  • An object can be another HTML file, JPEG image, Java applet, audio file, …
  • Each object is addressable by a URL (Universal Resource Locator)

HTTP: HyperText Transfer Protocol

  • Web’s application layer protocol
  • Client/server model
    • Client: browser that requests, recieves, and displays web objects
    • Server: web server sends objects in response to requests
  • HTTP/1.0: non-persistent connection
  • HTTP/1.1: persistent connection
    • May also use pipelining

More on HTTP

  • Uses TCP:
    • Client initiates TCP connection (creates socket) to server, port 80
    • Server accepts TCP connection from client
    • HTTP messages (application-layer protocol messages) exhanged between browser (HTTP client) and web server (HTTP server)
    • TCP connection closed
  • HTTP is “stateless”
    • Server maintains no information about past client requests
  • Aside: protocols that maintain “state” are complext
    • Past history (state) must be maintained if server/client crashes, their views of “state” may become inconsistent

HTTP Request MEssage

  • Two types of HTTP messages: request, response
  • HTTP request message:
    • ASCII (human-readable)
    • Request line: method, url, veresion, \r, \n
      • “GET /index.html HTTP/1.1\r\n”
    • Header lines
    • Blank line at the end indicates end of header

Method Types

  • HTTP/1.0
    • GET
    • POST
    • HEAD
      • Requesting the header only (i.e. response does not include the requested object)
  • HTTP/1.1
    • GET, POST, HEAD
    • PUT
      • Uploads file in entity body to path specified in URL field
    • DELETE
      • Deletes file specifie in the URL field from the server
    • And a few others
      • See te protocol specification in RFC2616

HTTP Response Message

  • Status line
    • “HTTP/1.1 200 OK\r\n”
  • Header lines
  • Blank line indicating end of header
  • Data, e.g., requested HTML file

HTTP Response Status Codes

  • Appears in the first line in server -> clinet response message
  • A few sample codes
    • 200 OK
      • Request succeeded, requested object later in this message
    • 301 Moved Permanently
      • Requested object moved
    • 400 Bad Request
    • 404 Not Found
      • Requested document not found
    • 505 HTTP Version Not Supported
